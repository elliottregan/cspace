# SmolVM feasibility evaluation

Date: 2026-04-23
Status: Evaluation — no implementation recommended at this time.

## Context

cspace's substrate is Docker + docker-compose. In a single afternoon (2026-04-22 → 23) we shipped four patch releases (v0.10.0 → v0.10.3) chasing compose ergonomic failures: missing embedded assets, a `PROJECT_ROOT` env-var collision, an OrbStack runc mount race, and auto-layering of a standalone devcontainer compose. The pattern is clear: we're reimplementing AI-agent sandboxing primitives on a substrate that was never designed for it.

[SmolVM](https://github.com/CelestoAI/SmolVM) is a purpose-built AI-agent sandbox using Firecracker (Linux) / QEMU (macOS) microVMs. Before investing in migration, we want a written evaluation focused on the features cspace actually depends on.

### MUST-HAVEs (user-defined)

1. **Web UI hosting** — dev server (Vite-class: HMR, WebSockets) inside each agent's sandbox.
2. **Friendly-URL host access** — reachable at `dev.<instance>.<project>.cspace.local`, not a random localhost port.
3. **Playwright in isolation per session** — each agent gets its own Chromium, no profile/cookie bleed.
4. **Per-session database** — each agent has its own disposable DB.
5. **DB reachable from both host and sandbox** — devtools on the host can connect, and the app running inside the sandbox can connect.
6. **Strong session isolation** — hardware-level boundary between agents.

---

## 1. Web UI hosting (Vite / HMR)

**cspace today.** The Vite dev server binds to `:3000`/`:4173` inside the container (`lib/templates/docker-compose.core.yml:108-109`). Host ports are Docker-assigned dynamically (`internal/ports/ports.go:20-28` — planet-named instances get `0` = Docker-assigned), then queried later for display.

**SmolVM.** Per-VM networking on macOS uses QEMU + SLIRP user-mode networking ([`docs/deep-dive/qemu-guest-networking.md`](https://github.com/CelestoAI/SmolVM/blob/main/docs/deep-dive/qemu-guest-networking.md)). Host → guest port forwarding is `hostfwd` rules declared at VM creation time, e.g. `hostfwd=tcp:127.0.0.1:2200-:22`. No documented WebSocket-specific path; SLIRP generally proxies TCP transparently, so HMR should work but hasn't been verified for Vite.

**Gap.** 🔧 Adapter. We'd add `--hostfwd` entries at `smolvm create` time for every port we know about in advance. Vite's default `5173` is fine; ad-hoc ports opened later would require recreating the VM.

**Cost.** `1-3 days` including HMR / WebSocket validation against the SLIRP path.

---

## 2. Friendly-URL host access

**cspace today.** Traefik runs as a shared proxy (`internal/docker/docker.go:145 EnsureProxy`). Each cspace container declares Traefik labels (`lib/templates/docker-compose.core.yml:37-45`, e.g. `rule=Host(\`dev.${CSPACE_INSTANCE_NAME}.${CSPACE_PROJECT_NAME}.cspace.local\`)`). `*.cspace.local` resolves via a `cspace-dns` sidecar to Traefik's Docker IP, both on the host (`/etc/hosts` or resolver) and inside containers (`internal/docker/docker.go:290 InjectHosts`).

**SmolVM.** No hostname routing primitive. Exposes only `hostfwd=tcp:127.0.0.1:<port>-:<port>` style bindings per VM. The browser "live view" at `:6080` is a plain port on 127.0.0.1 (example repo [`examples/browser_session.py`](https://github.com/CelestoAI/SmolVM/blob/main/examples/browser_session.py) exposes `session.live_url`). No mention of hostname routing, Traefik integration, or DNS service discovery in README or `docs/`.

**Gap.** ⚠️ Build on top. We'd keep Traefik on the host, translate from `<friendly>.<instance>.<project>.cspace.local` → `127.0.0.1:<hostfwd-port-for-VM>`, and emit hostfwd rules per exposed service at VM create time. The cspace hosts injection (`InjectHosts`) doesn't help inside VMs — they have their own 10.0.2.x network and can't see the host Traefik via `cspace.local`.

**Cost.** `1-3 days` for a host-side Traefik + Go-to-Traefik dynamic config + the port-mapping bookkeeping.

---

## 3. Playwright / browser per session

**cspace today.** A separate `browser` service per cspace instance (`lib/templates/docker-compose.core.yml:119-143`) runs Chrome CDP on `:9223` and Playwright run-server on `:3000`, inside an `mcr.microsoft.com/playwright:v1.58.0-noble` image. Profile isolation is implicit — each container has its own filesystem, so per-instance browsers never share a Chromium profile directory.

**SmolVM.** Native primitive. [`examples/browser_session.py`](https://github.com/CelestoAI/SmolVM/blob/main/examples/browser_session.py) shows `BrowserSession(BrowserSessionConfig(mode="live", record_video=True, ...))` producing `session.live_url` (noVNC) and `session.cdp_url` (Playwright-compatible CDP). Each session is a separate VM → hardware-isolated profile, no bleed possible by construction.

**Gap.** ✅ Native. This is the one slot where SmolVM is clearly better than what cspace does today — isolation is kernel-level rather than container-level, and the live-view is built in instead of requiring separate noVNC/socat plumbing.

**Cost.** `<1 day` to wire cspace's existing MCP consumers (chrome-devtools-mcp, playwright-mcp) to the `cdp_url`. Potentially saves the whole Chrome-in-a-container + socat + `init-firewall.sh` bypass dance.

## 4. Per-session database

**cspace today.** Project-configurable. The Convex pattern (see `/Users/elliott/Projects/resume-redux/.devcontainer/docker-compose.yml`) runs `convex-backend` as a sibling container on the project Docker network, addressable by service alias `http://<container>:3210`. Per-instance DB is the common case; currently per-*project* because the Convex service is in the project network, not the per-instance network.

**SmolVM.** No documentation of running databases inside VMs or managing DB-sidecar VMs. No examples. Available options:

- **DB inside the agent VM.** Run Postgres/Convex as a local process. Works, but bloats the image and loses isolation between DB and agent code (same kernel). Also breaks the "reachable from host" requirement unless we forward the DB port too.
- **DB as sibling VM.** Matches cspace's current model. **Blocked on macOS** by SLIRP: each VM has its own user-mode network, VMs cannot address each other directly. TAP-based VM-to-VM networking exists on Linux (mentioned in `docs/deep-dive/optimization-plan.md` — "IP pool /16 expansion"), but SmolVM on macOS is SLIRP-only (`docs/installation.md` — "On macOS, SmolVM automatically uses the QEMU backend" + the SLIRP doc).
- **DB on the host.** Possible. Agent reaches it via the SLIRP gateway (`10.0.2.2`). Loses per-session isolation unless we run N databases on the host keyed by instance.

**Gap.** 🔧 / ⚠️ depending on shape. In-VM DB: `1-2w` (image build + Convex init scripts + port forwarding). Sibling VM on macOS: ⛔ blocked today.

**Cost.** `1-2w` for in-VM approach; deal-breaker for cross-VM approach on macOS until SmolVM ships bridged networking.

---

## 5. DB reachable from host and sandbox

**cspace today.** Host → container via Traefik Host rule (e.g. `convex.<instance>.<project>.cspace.local` → service port); container → sibling container via Docker network alias.

**SmolVM.** Host → VM via `hostfwd`, one port per mapping, declared at VM create. VM → host via the SLIRP gateway IP `10.0.2.2`. VM → VM: blocked on SLIRP (macOS). No hostname routing — only raw ports.

**Gap.** 🔧 for the simple case (DB in the same VM as the agent — host reaches via hostfwd, "sandbox" is the same VM so trivially local). ⛔ if we need a separate DB VM reachable from the agent VM on macOS.

**Cost.** `<1 day` if DB lives in the agent VM; otherwise blocked.

---

## 6. Session isolation

**cspace today.** Container-level: shared kernel, namespace-based isolation. Adequate for honest agents, weak against compromise.

**SmolVM.** Firecracker (Linux) / QEMU + Hypervisor.framework (macOS). Full kernel per VM. Hardware isolation. This is the headline win — `--dangerously-skip-permissions` runs in a real VM boundary instead of a shared-kernel namespace.

**Gap.** ✅ Native, strictly better than current.

**Cost.** Free (automatic property of the substrate).

## Cross-cutting concerns

### Image build pipeline — ⚠️ significant gap

cspace's image (`lib/templates/Dockerfile`) bakes Node + Claude CLI + git + pnpm + Playwright deps + firewall scripts + supervisor. Standard Dockerfile, cached layers, `cspace rebuild` for iteration.

SmolVM uses ext4 rootfs images ([`docs/deep-dive/filesystem-and-disk-images.md`](https://github.com/CelestoAI/SmolVM/blob/main/docs/deep-dive/filesystem-and-disk-images.md) — "Both Firecracker and QEMU use ext4 as the root filesystem format"). There is **no documented Dockerfile-equivalent**, no `smolvm build` command, no OCI import path. The image build helper mentioned (`build.py` using `mkfs.ext4`/`mke2fs`) is internal, undocumented for external users. We'd build our own pipeline: debootstrap → chroot install → mkfs.ext4. Probably 1-2 weeks to get parity with the current Dockerfile.

### Host filesystem sharing — ⛔ deal-breaker for `.cspace/context`

cspace relies on bidirectional bind-mounts for live inter-agent state. Specifically, `.cspace/context/` (the cspace-context MCP server's store) is mounted into every container; when agent A writes a finding, agent B sees it immediately without git push/pull. This is the core value proposition of the context MCP server (see `docs/superpowers/specs/2026-04-13-context-mcp-server-design.md`).

SmolVM mounts (via `--mount`, [issue #157](https://github.com/CelestoAI/SmolVM/issues/157) closed 2026-04-13) are **read-only with an overlay for writes** — writes stay inside the VM and do NOT propagate back to the host. Verbatim from the maintainer:

> "We just added folder mounting. Note that this is read-only for now. Any change that you perform inside the VM would not travel back to the original host files."

Additionally, mounts use virtio-9p and are **QEMU-only, not Firecracker/KVM** ([issue #178](https://github.com/CelestoAI/SmolVM/issues/178), open).

Workarounds:
- Rewrite the context MCP server to talk over the network (host-side daemon, agents reach it via SLIRP gateway). Abandons the "live file watching" simplicity of the current design.
- Wait for bidirectional mounts (no ETA, and the feature request note suggests they're not actively pursuing it).

### Network egress allowlist — platform-split

cspace's `init-firewall.sh` runs iptables *inside* the container, resolves domain list, drops outbound traffic not in allowlist.

SmolVM's egress control ([`docs/concepts/network-egress-controls.md`](https://github.com/CelestoAI/SmolVM/blob/main/docs/concepts/network-egress-controls.md)) uses host-side nftables per TAP device. Explicit caveat: "SLIRP incompatibility — host-side nftables rules don't see it." On macOS (SLIRP only) we lose the host-enforced allowlist and would need guest-side iptables, i.e. exactly what cspace does today. No improvement on macOS.

### Go ↔ Python API — adapter tax

SmolVM ships a Python CLI + Python SDK. No HTTP/gRPC daemon surface documented. cspace is Go. Integration path is shelling out to `smolvm` CLI from Go (`exec.Command("smolvm", ...)`). Loses type-safe config, harder to represent the full networking model, and we'd be passing arguments we don't fully understand through a subprocess. Workable but degrades the codebase's current ergonomics.

### Maturity / bus factor

- 411 GitHub stars.
- Latest release `v0.0.10` (2026-04-19).
- 17 releases. Repo created recently; active but small surface area of contributors.
- One vendor (Celesto AI).
- Apache-2.0, which is fine.
- Issue response is prompt (#157 added `--mount` within days of the request), which is a positive signal but also implies the roadmap is reactive, not hardened.

A substrate migration is a multi-year bet. Running cspace on a single-vendor, v0.0.x project with 411 stars is a real risk the evaluation must call out.

### macOS specifics

cspace's primary users are on macOS + OrbStack. On macOS SmolVM is **QEMU, not Firecracker** (explicit in `docs/installation.md`). The headline "Firecracker microVMs, ~500ms boot" numbers are Linux/KVM; QEMU + Hypervisor.framework is slower to boot (likely 1-3s) and has different syscall / networking characteristics. Mounts work (QEMU-only feature). VM-to-VM networking does not (SLIRP-only). Egress nftables does not (SLIRP-only).

The macOS path is materially worse than the Linux path on almost every axis except the baseline "real VM isolation" guarantee.

## Recommendation: **Stay on Docker.**

Primary driver: the `.cspace/context` live-sharing requirement is a **⛔ today** on SmolVM (RO mount only, upstream has no ETA on bidirectional support). That alone breaks the coordinator/worker collaboration model the context MCP server was built for.

Secondary driver: on macOS (the user's primary platform) SmolVM degrades to QEMU + SLIRP, which disables VM-to-VM networking and host-side egress filtering — two of the features that would have been clear wins on Linux. A migration optimized for macOS gets the worst of both platforms.

Tertiary: migrating to a v0.0.10, single-vendor, 411-star project to escape Docker's maturity and ecosystem cost is a trade that only makes sense if the gains are overwhelming. They aren't.

**What we do instead (cheap wins):**

1. **Single generated compose file per instance.** Collapse core + search + project-paths-override + auto-detected devcontainer into one Go-rendered YAML per `cspace up`, with absolute paths throughout. Kills the whole class of bugs we hit this week (`PROJECT_ROOT` collision, multi-`-f` interpolation ambiguity, OrbStack `include:` resolution quirks, standalone-devcontainer auto-layering). Estimated 1-2 days.
2. **Document the `cspace:` service contract** for auto-layered devcontainer compose files (landed in v0.10.3, but not yet written up in `docs/`).
3. **Keep an eye on SmolVM.** Revisit in 6 months if they ship: bidirectional host mounts, bridged VM-to-VM networking on macOS, a documented custom-image build path, and a Go binding or stable HTTP API. Those four unblock roughly everything in this evaluation.

### Conditions that would flip the recommendation

- SmolVM ships bidirectional host-fs mounts → context sharing no longer blocked.
- SmolVM ships bridged VM networking on macOS → per-session sibling-VM databases work.
- The cspace threat model changes to require hardware-level isolation (e.g., we start running third-party agent code, not just Claude on our own repos) → Docker's shared-kernel isolation becomes inadequate regardless of other costs.

## Open questions (for a future spike)

If any of the "flip" conditions land, a 1-week spike should answer:

1. **Vite HMR over SLIRP** — does WebSocket framing survive user-mode networking at dev speed, or do we need to switch to polling? Simple reproducer: create a SmolVM, run Vite inside it with `hostfwd=tcp:127.0.0.1:5173-:5173`, observe HMR behaviour.
2. **Rootfs build time** — how long does it take to build a Node + Claude CLI + Playwright ext4 image on macOS using `debootstrap` + chroot? Target <2 minutes for `cspace rebuild` to feel like today.
3. **Multi-VM boot cost** — running 4 agents concurrently today = 4 Docker containers ≈ 500MB RAM. SmolVM with 4 VMs each with their own kernel = ~2GB minimum. Is that acceptable on a 16GB MacBook?
4. **Playwright live-view UX** — does `session.live_url` (noVNC) beat the current Traefik-routed CDP? Quick qualitative check.
5. **Go subprocess friction** — prototype a `cspace up` that shells out to `smolvm create --hostfwd ... --mount ...` and measure whether the config surface is expressible.
