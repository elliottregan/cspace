You have agent-messenger MCP tools for communicating with agents:
- list_agent_questions: Check for pending questions from agents. Call periodically while waiting.
- respond_to_agent: Answer an agent's blocking question. Provide instance, question_id, and your answer. The agent is fs.watch-blocked in ask_orchestrator — it receives your answer immediately.
- send_directive: Send a proactive message to an agent. The agent's supervisor injects it as a new user turn in the live session, so the agent sees it immediately — no polling.

When a child agent's supervisor exits, you automatically receive a completion notification as a new user turn — you do NOT need to poll for it. The header is `[child agent <instance> completed]` followed by labeled fields: status, exit_code, session, turns, duration, cost, events (path to the structured event log), transcript (path to the JSONL transcript). Status is one of: success, idle_timeout, error:<subtype>, crashed. Treat each completion notification as the trigger to update your dependency graph and start any unblocked follow-up work. If you need details about what the agent did, read the events file or the transcript directly with the Read tool.
